
# 系统导航 — AI 的信息路由表

> 这份文件回答一个根本问题：**你的 AI 在什么时候该读什么。**
>
> 它不是功能清单。它是触发系统。什么信号 → 加载什么模块。
> 我们不给现成的路由表——你的模块、你的信号、你的场景，只有你知道。

---

## 为什么需要系统导航

当一个 AI 的能力模块变多，你不能让它每次启动把全部东西加载到脑子里。

那样它会被淹没。上下文是有限的。无关信息挤走关键判断。

解决方案：**平时只加载最核心的东西。触发了才加载。**

就像你不会把整本操作手册带在身上——你带目录。翻到哪页，看哪页。

---

## 设计原则

### 1. 分层，不堆砌

```
永远层 —— 每次启动必读。量极少，只放最核心的。
路由层 —— 某种"信号"出现时才加载。
精准层 —— 极少用的深度模块，需要时才翻。
```

### 2. 触发条件要具体

不是"需要分析的时候加载"。是"当用户提到[关键词]或任务涉及[具体场景]时加载[具体文件]"。我的触发条件，有时候会用引导推演的方式。

模糊的触发条件 = 要么不触发，要么乱触发。

### 3. 兜底机制

触发信号没命中任何模块 → 怎么办？你的 AI 不该愣住。它该知道找谁（不知道怎么判断看➡️ceo.md，甚至设计判断兜底机制.如果依然无法命中你还可以➡️”价值判断触发“，这个时候，如果它依然不知道怎么办，那就不是它的问题，我们可以和他一起想办法把这件事情给解决掉➡️问你。）

## 结构


### 一、永远层

这些文件永远在上下文中。不算多——几个文件就够了。
只有灵魂、边界和路由表（Agents.md在“路由表”中）。
我们采用的是按需加载，包括技能skill

| 文件                   | 它知道           |
| -------------------- | ------------- |
| [你的 SOUL.md]         | 我是谁           |
| [你的 CONSTITUTION.md] | 边界在哪          |
| 路由表（自己取名字.md）        | 我知道这条指令往哪里走   |
| [你的方向/规划文件CEO.md]    | 现阶段要达成什么,心里有底 |
|                      |               |

### 三、路由触发表

这是整个系统的核心。

对每一种任务场景，定义触发信号和对应的模块：

| 触发信号（关键词或场景） | 加载什么 | 备注 |
|------------------------|---------|------|
| [你的场景 A] | [对应的模块文件] | [需要注意什么] |
| [你的场景 B] | [对应的模块文件] | |
| [你的场景 C] | [对应的模块文件] | |
| [异常/错误/失败] | [故障处理模块] | |
| [安全问题/攻击] | [安全应急模块] | |
| [以上都未命中] | [兜底处理方式] | |

> 不用一次写完所有场景。先把最常见的 3-5 个写好，后续你会自然发现"还缺"。

### 四、模块清单

把你所有的能力模块列在这。这是一份索引——不是每次都要加载的。

| 分类 | 模块文件 | 什么时候读 |
|------|---------|:--:|
| 核心 | [你的核心模块] | 启动 |
| 工作流 | [你的工作流模块] | 按路由触发 |
| 规则 | [你的规则模块] | 启动 + 按场景 |
| 记忆 | [你的记忆模块] | 启动读索引 · 决策时查 |
| [你的其他分类] | [路径] | [条件] |
